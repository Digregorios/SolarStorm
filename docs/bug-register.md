# Bug Register — Known Issues and Fixes

## Resolved

### #1: METAR regex missing end-of-string anchor
- **Date:** 2026-06-04
- **Commit:** `54079fe`
- **Symptom:** Temperature group at end of METAR string not matched when it was
  the last token before newline
- **Root cause:** Regex `r"\s(M?\d{1,2})/(M?\d{1,2})\s"` required trailing
  whitespace; end-of-string has none
- **Fix:** Added `(?=\s|$)` lookahead: `r"\s(M?\d{1,2})/(M?\d{1,2})(?=\s|$)"`
- **Lessons:** Defensive regex for line-oriented formats. Always test with
  real METAR strings at end-of-buffer.

### #2: CP hours interpreted as local time instead of UTC
- **Date:** 2026-06-05
- **Commit:** `139a0e4`
- **Symptom:** CP-to-UTC conversion was treating `20:00` as local time,
  producing wrong UTC timestamps for feature cutoff calculations
- **Root cause:** METAR convention uses UTC throughout; CP times in the
  Polymarket contract are also UTC
- **Fix:** `cp_to_utc` now treats CP hour as UTC hour on the given local date,
  with dual-candidate search (try date, then date-1) for DST-safe mapping
- **Lessons:** Always clarify UTC vs local in meteorological time handling.
  Contracts should specify timezone explicitly.

## Open

### #3: Persistence baseline MAE=0 for late CPs (fixed with aggregation)
- **Date:** 2026-06-05 (fixed)
- **Symptom:** Leaderboard showed persistence with MAE=0.00 on every day
- **Root cause:** Per-row `LadderResult` with `n=1` stored raw abs(error).
  For late CPs where k_cp already equals Tmax, single-row error is 0.
  Leaderboard displayed 232 individual rows instead of 16 aggregated summaries.
- **Fix:** `aggregate_results()` function pools per-row errors by
  `(level, name, cp)` and computes proper MAE, RMSE, bias, bracket_match.
  Leaderboard now shows 4 baselines × 4 CPs = 16 aggregated entries.
- **Lessons:** Never display per-row metrics as if they were aggregated.
  Always compute MAE as `mean(abs(errors))`, not per-row.

### #4: Empirical conditional 100% fallback rate
- **Date:** 2026-06-05 (diagnosed, not fixable)
- **Symptom:** L4 empirical_conditional shows 100% fallback and MAE=7.96
  on recent 28-day window
- **Root cause:** Structural data sparsity. 616 (month, cp, k_cp) buckets
  across 4491 complete days. 53.6% of buckets have fewer than 30 samples
  (below `n_min_bucket` threshold). Average ~7.3 days per bucket.
- **Verdict:** NOT A BUG — this is a fundamental limitation of the
  nonparametric conditional approach. The model is working as designed
  (falling back to marginal distribution). Use for benchmarking only,
  never for production forecasting.
- **Lessons:** `n_min_bucket=30` with 12×4×~15=720 possible buckets and
  ~4500 days means average fill rate of ~6 days/bucket. Conditional
  empirical is structurally sparse for this feature space.
