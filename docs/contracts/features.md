# Feature Contracts

The formal feature contract lives in the validated_feature_contract.json output of the `validate` command, generated each run to `reports/YYYY-MM-DD/validated_feature_contract.json`.

This document provides context and explains the contract's structure, evolution, and relationship to the legacy Wellington contracts.

## Historical Origin

The contracts in `quarentena/Wellington/contracts/` (Phase 1 baseline, frozen 2026-05-29) defined the old project's feature schema. Key documents:

- **features.md:** Feature inventory v0.1 with hard rules (causal timestamp, closed='left', no `tmpf` for `ts >= cp_utc`).
- **objective.md:** Trading objective function v1.0 (max EV, subject to drawdown/coverage/trade/spike constraints).
- **imputation.md:** Missing value handling rules (carried forward into SolarStorm's `quality` tag system).
- **quantization.md:** Temperature binning rules (superseded by SolarStorm's decimal-internal integer-output design).
- **resolver.md:** Contract resolution rules (superseded by `_settlement.py`).

## Current Contract Structure

The `validated_feature_contract.json` is generated by `validate_hypotheses()` and contains:

```json
{
  "validated_features": [
    {
      "id": "H1",
      "feature_column": "slope_3h",
      "cp": "20:00",
      "regime": "all",
      "effect_size": 0.123,
      "ci_lo": 0.045,
      "ci_hi": 0.201,
      "p_value": 0.002
    }
  ],
  "blocked": [
    {
      "id": "H19",
      "feature_column": "sst_maritime_cap",
      "reason": "Requires Cook Strait SST -- no METAR source available (H19)"
    }
  ],
  "generated": "2026-06-05T12:00:00+00:00",
  "alpha": 0.05,
  "n_hypotheses_tested": 88,
  "n_validated": 0,
  "n_rejected": 88
}
```

### Validated Features

A feature is **validated** when:
1. Walk-forward bootstrap 95% CI excludes zero (CI low > 0.0).
2. Benjamini-Hochberg FDR correction at alpha=0.05 survives.
3. All five gates (G1-G5) pass.

Each validated entry includes:
- `id`: Hypothesis ID (H1-H23)
- `feature_column`: Column name in `features.parquet`
- `cp`: Checkpoint hour (20:00, 21:00, 22:00, 23:00)
- `regime`: "all" or a specific regime (calm, transition, late_warming, foehn_nw, disrupted)
- `effect_size`: Mean error reduction vs baseline (positive = better)
- `ci_lo`, `ci_hi`: 95% bootstrap confidence interval bounds
- `p_value`: One-sided bootstrap p-value for H0: mean(baseline) <= mean(challenger)

| `hours_to_expected_peak`      | H2   | Hours from CP to climatological Tmax time            |
| `dewpoint_depression`         | H4   | Mean (T - Td) at CP                                  |

| `tmin_delta_tmax`             | H6   | Tmin - Tmax(D-1)                                     |

| `wind_dir_change_s_to_n`      | H8   | Wind direction shift S -> N (degrees)                |

## Contract Versioning

The feature contract is regenerated each time `tmax validate` runs. Features that pass validation in one run may fail in a later run if:
- More training data becomes available (changing the walk-forward splits).
- Feature computation logic changes.
- New baselines shift the null comparison.

The contract is versioned by the generation timestamp in the `generated` field.

## Comparison with Legacy Contracts

The old project's `features.md` contract (v0.1, frozen 2026-05-29) defined 18 features organized by source. SolarStorm carries forward the hard rules (causal timestamp, closed='left', no future leakage) and the feature inventory concept, but:

- Feature IDs are linked to testable hypotheses (H1-H23), not a static inventory.
- Features are validated via walk-forward bootstrap CI, not design review.
- The contract is machine-generated (JSON), not a static markdown document.
- Blocked features are explicitly tagged with reasons.

The legacy contracts remain in `quarentena/Wellington/contracts/` for historical reference.
